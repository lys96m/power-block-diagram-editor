# ADR 0003: UI コンポーネントカタログ導入の検討

## コンテキスト

- コンポーネント数が増え、分割も進んでいるため UI 振る舞いを単体で確認できる仕組みがあると開発効率が上がる。
- Storybook を導入する場合、開発環境が増えることによるセットアップと CI コストが発生する。

## 選択肢

1. Storybook を導入し、主要コンポーネント（PropertiesPanel, NetManager, Node/Edge フォームなど）をカタログ化する。
2. 画面内で簡易な「デモページ」を追加し、実アプリの一部として UI を確認する。
3. 現状維持（優先度低のため、ドキュメントに将来案として残す）。

## 決定

- 現時点では **3. 現状維持** とし、Storybook 導入は見送る。理由:
  - テストスイートにスモークテストが追加され、最小限の UI 回帰がカバーできている。
  - 優先度はドメイン機能の充実・検証強化に置き、カタログ導入は後続の拡張時に再検討する。
- 将来、デザインバリエーションやコンポーネント数がさらに増えた場合は Storybook 導入を再検討する。

## 影響

- 新規ツール導入は行わないため、ビルド/CI への影響はなし。
- 必要性が高まったら、この ADR を見直し、導入手順と運用ルールを別途定義する。

## メモ

- 再検討のタイミング: 大きな UI 変更が続く、あるいはデザイナー/QA がコンポーネント単位での確認を求める場合。
